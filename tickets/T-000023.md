## Goal
Implement three search strategies (exploit best config, explore alternative configs, random) to avoid local maxima in the autoresearch loop

## Acceptance Criteria
- [ ] `random_config()` function generates a fully random SolverConfig
- [ ] `explore_config(best)` function picks a different base region than the current best (e.g. different strategy) and mutates from there
- [ ] `mutate_config(best)` remains the exploit path (unchanged)
- [ ] Experiment loop allocates a configurable mix (default: 60% exploit, 20% explore, 20% random)
- [ ] Each experiment is tagged with its search strategy (exploit/explore/random) in output and ledger
- [ ] All three strategies can discover and commit new all-time bests

## Notes

### The Problem
Current autoresearch is pure hill-climbing: mutate the best config by one knob.
This gets stuck in local maxima — once you find a decent config, small mutations
can't escape to a potentially better region of the search space.

### Three Strategies

1. **Exploit** (current behavior): `mutate_config(best)` — change one knob from
   the current best. Greedy, efficient, but converges to local optima.

2. **Explore**: Start from a deliberately different base config (different strategy
   backend, opposite propagation setting, etc.) and mutate from there. Searches
   regions the exploit path hasn't visited.

3. **Random**: `random_config()` — every knob is independently randomized. No bias
   toward any known good config. Unlikely to beat the best on any given trial, but
   occasionally discovers surprising combinations.

### Mix Allocation
Default: 60% exploit, 20% explore, 20% random (configurable via CLI args).
For a 20-experiment run: ~12 exploit, ~4 explore, ~4 random.
